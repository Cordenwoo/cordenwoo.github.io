Git学习指南笔记-01

2020-11-18

为什么要用Git?

  • Git允许我们利用分支来开展工作
  • 工作流上的灵活性：不但单一开发者可以使用它，敏捷团队也可以找到使用它的合适方法。
  • 适合奉献工作：如果我们使用Git，每个人都可以克隆一个独立的版本库，然后再对其进行后续的改动。
  • 高性能
  • 强大的抗故障和抗攻击的能力
  • 离线开发与多点开发
  • 强大的开源社区
  • 可拓展性：Git为用户提供了许多实用命令，其中包括了能是我们更便于直接访问其远程版本库的命令。

分支与合并是Git中两个强大的功能。但是，我们应该开多少个分支呢？是根据每个软件功能来开？还是正对每个发行版本来开？还是只该有一个分支？

版本库，分布式工作的基础所在

版本库的本质上就是一个高效的数据存储结构而已，由以下部分组成：

  • 文件（blob）
  • 目录（Tree）
  • 版本（commit）

在Git中，我们可以开启有针对性的分支，即显式地创建一个分支。显式分支通常主要用于协调某一种功能性的开发。

准备Git环境

可以宣布用 config 命令配置一下用户名和用户邮箱：

git config --global user.emall "*********"
git config --global user.name "YourName"

创建一个Git项目

进入项目的目录，创建一个版本库，用于储存项目本身及历史。我们需要在该项目的目录中使用 init 命令。对于一个带版本库的项目录，我们通常称之为工作区。

cd /yourproject
git init

init 命令会在这个目录中创建一个名为 .git 的隐藏目录，并在其中创建一个版本库

工作区与版本库 ：工作区是一个包含.git的子目录（内含版本库）中的目录。我们可以用 init 命令在当前目录中创建

如何提交

在Git中，我们通常将项目的一个版本称之为一次提交，但这需要分为两个步骤来实现

  1. 先用 add 命令来确定哪些文件应该被包含在下次提交中（把文件提交到版本库）
  2. 用 commit 命令将修改传送到版本库中，并赋予该提交一个Message
git add filename.xxx
git commit --message "Message content."

我们要对修改过的文件或者新文件执行 add 命令，对要删除的文件使用 rm 命令

检查状态

git status

status 会显示出该项目自上次提交以来所发生的所有修改。有可能有文件被表示成了未跟踪状态（untracked files)，这是因为没有用 add 命令将其注册到版本库。

显示历史

log 命令可以用来显示项目的历史，所有提交都会按时间顺序被降序排列出来。 git log

Git的协作功能

克隆版本库

一个新的开发者需要一个属于他自己的版本库副本（也称之为克隆体）。该副本包含了所有的原始信息与整个项目的历史信息。

git clone /xxxx /xxxx-clone

以xxxx-clone为名字克隆xxxx，/xxxx为项目的url，/xxxx-clone也是url，表示在哪个目录下新建目录

从另一个版本库中获取修改

使用 pull 命令将另一个版本库中的新提交传递过来。理解为 拉取 。由于在创建克隆版本库时，原版本库的路径就已经被储存在了它的克隆体中，因此 pull 命令知道该从哪里去取回重新的提交。

cd /xxxx-clone 
git pull

在这个过程当中， pull 命令从原版本库中取回了新的修改，将它们与克隆体的本地修改进行了对比，并在工作区中合并了两边的修改，创建了一次新的提交。这个过程叫做 合并（merge）

在没有参数的情况下， pull 命令只能在克隆版本中发挥作用。当然也可以用参数来置顶任意版本库的路径。

cd /xxxx-clone 
git pull /xxxx master

创建共享版本库

除了可以使用 pull 命令从其他版本库中取回相关提交外，我们也可以用 push 命令将提交传送给其他版本库。只不过， push 命令只适用于那些没有开发者在上面开展具体工作的版本库。（可以联想到 Github 就是托管这种版本库的平台）。用来充当开发者们传递提交（使用 push 命令的汇聚点），以便其他人可以从中拉回她们所作的修改。

如何创建没有开发者工作的版本库——创建一个不带工作区的版本库，称之为 裸版本库（bare repository） 。可以使用 clone 命令的 –bare 选项来创建一个裸版本库。

git clone --bare /xxxx /xxxx-clone-bare.git

以xxxx-clone-bare.git为名字克隆xxxx。xxxx-clone-bare.git 该目录只能用来交换提交。它本身没有工作区

在我们做完修改后，就可以用 push 命令向共享版本库传送该提交了。该指令的的参数要求与 pull 相同，我们需要指定目标版本库的路径及分支。

git push /xxxx /xxxx-clone-bare.git master
